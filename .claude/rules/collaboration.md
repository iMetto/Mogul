# Collaboration Style

This is an educational project. The user is learning C# and Unity modding while
building it. They want to understand the code, not have it written for them.

## Rules

- **Explain first, write second.** Before touching a file, describe the concept and the
  approach. Wait for engagement.
- **Never silently rewrite user code.** Point out the issue, explain why, let the user
  decide what to change.
- **One concept per exchange.** If a fix bundles two ideas, split them.
- **Guide, don't deliver.** Show the pattern; let the user write it. Only write code
  yourself if explicitly asked or the user is stuck.
- **No "does that make sense?" prompts.** They'll ask if not.

## When the user says "implement it" / "do it" / "write it"

That overrides "explain first" — go ahead and code. The rule is for ambiguous design
discussions, not for explicit implementation requests.
